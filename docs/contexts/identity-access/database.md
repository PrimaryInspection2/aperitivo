# Identity & Access — Database Schema

Physical schema for the IAM Bounded Context: tables, columns, constraints, indexes, and migration notes. Derived directly from the [domain model](domain-model.md).

Conventions: snake_case identifiers; UUID primary keys; `timestamptz` for all instants (stored UTC); migrations are Flyway per-BC changesets applied at startup.

## Schema ownership

IAM owns its own logical schema within the shared PostgreSQL instance. Other BCs never read these tables directly — they go through IAM's published interfaces or events (see [ADR 0005](../../adr/0005-bounded-contexts.md)). Concretely this is the `iam` schema (or table prefix, depending on the migration tool's layout); the DDL below omits the schema qualifier for readability.

## Tables

### users

```sql
CREATE TABLE users (
    id            UUID        PRIMARY KEY,
    display_name  TEXT        NOT NULL,
    email         TEXT        NULL,
    avatar_url    TEXT        NULL,
    created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

Notes:
- `id` is our internal UUID, generated by the application (not a DB sequence), so it is known before insert and usable as the JWT `sub` and as the FK target everywhere.
- No provider-specific columns by design — see invariant 1 in the domain model. `email` and `avatar_url` come from the Strava profile and are nullable (Strava may not expose email depending on granted scopes).
- No `status` column. A user's existence is binary; connection health lives on `connected_sources`.

### connected_sources

```sql
CREATE TABLE connected_sources (
    id                       UUID        PRIMARY KEY,
    user_id                  UUID        NOT NULL,
    provider                 TEXT        NOT NULL,          -- enum as string: 'STRAVA', …
    provider_user_id         TEXT        NOT NULL,          -- e.g. strava_athlete_id
    access_token             BYTEA       NOT NULL,          -- AES-GCM ciphertext (nonce||ct||tag)
    refresh_token            BYTEA       NOT NULL,          -- AES-GCM ciphertext
    access_token_expires_at  TIMESTAMPTZ NOT NULL,
    scopes                   TEXT[]      NOT NULL DEFAULT '{}',
    status                   TEXT        NOT NULL,          -- 'ACTIVE' | 'REVOKED' | 'ERROR'
    failure_count            INT         NOT NULL DEFAULT 0,
    last_error               TEXT        NULL,
    last_refreshed_at        TIMESTAMPTZ NULL,
    last_used_at             TIMESTAMPTZ NULL,
    version                  BIGINT      NOT NULL DEFAULT 0,  -- @Version optimistic lock
    created_at               TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at               TIMESTAMPTZ NOT NULL DEFAULT now(),

    CONSTRAINT fk_connected_sources_user
        FOREIGN KEY (user_id) REFERENCES users (id)
);
```

Notes:
- `user_id` is a **plain FK column**, not a JPA association — navigation is via repository methods (see the persistence policy in the domain model). The DB-level `FOREIGN KEY` still enforces referential integrity; that is independent of how the ORM maps it.
- `access_token` / `refresh_token` are `BYTEA`: the application encrypts with AES-GCM via `EncryptedBytesConverter`, storing `nonce || ciphertext || authTag`. The DB never sees plaintext.
- `provider` and `status` are stored as `TEXT` (matching `@Enumerated(EnumType.STRING)`), not native PG enums — easier to evolve (adding `GARMIN` is an application enum change, no `ALTER TYPE`).
- `scopes` is a native Postgres `TEXT[]`, mapped by the custom array type. Small fixed set; never queried element-wise, so an array beats a join table here.
- `version` backs Hibernate `@Version` optimistic locking, used to serialize concurrent token refreshes.

## Constraints and indexes

```sql
-- One identity per (provider, external id). Resolves login to an existing source.
ALTER TABLE connected_sources
    ADD CONSTRAINT uq_connected_sources_provider_identity
    UNIQUE (provider, provider_user_id);

-- At most one NON-revoked source per (user, provider), while still allowing
-- a fresh row after a revoke (reconnect). Partial unique index.
CREATE UNIQUE INDEX uq_connected_sources_active_per_user_provider
    ON connected_sources (user_id, provider)
    WHERE status <> 'REVOKED';

-- Hot lookup: resolve a user's active Strava source (TokenManager, ingestion).
CREATE INDEX ix_connected_sources_user_provider
    ON connected_sources (user_id, provider);

-- Reconciliation / proactive refresh sweeps: find ACTIVE sources by expiry.
CREATE INDEX ix_connected_sources_status_expiry
    ON connected_sources (status, access_token_expires_at);
```

Index rationale:
- `uq_…_provider_identity` enforces invariant 2 and is the lookup for "does a user already exist for this Strava athlete?" during login (`findByProviderAndProviderUserId`).
- `uq_…_active_per_user_provider` enforces invariant 3. Making it a **partial** index (`WHERE status <> 'REVOKED'`) is the key trick: a revoked row can coexist with a new active one, so reconnect after revoke does not violate uniqueness.
- `ix_…_user_provider` serves the hottest read — `findByUserIdAndProvider` on every token access. (It overlaps the partial unique index but is unconditional, so it also covers revoked rows for audit/history reads.)
- `ix_…_status_expiry` serves periodic sweeps: reconciliation (`findByStatus(ACTIVE)`) and any future proactive-refresh job ("ACTIVE sources expiring within the hour").

## Mapping notes (entity ↔ schema)

| Entity field | Column | Mapping detail |
|---|---|---|
| `accessToken: String` | `access_token BYTEA` | `@Convert(EncryptedBytesConverter)` — AES-GCM |
| `refreshToken: String` | `refresh_token BYTEA` | `@Convert(EncryptedBytesConverter)` — AES-GCM |
| `provider: Provider` | `provider TEXT` | `@Enumerated(EnumType.STRING)` |
| `status: SourceStatus` | `status TEXT` | `@Enumerated(EnumType.STRING)` |
| `scopes: String[]` | `scopes TEXT[]` | custom `StringArrayType` |
| `version: long` | `version BIGINT` | `@Version` |
| `createdAt/updatedAt` | `created_at/updated_at` | `@CreationTimestamp` / `@UpdateTimestamp` |

The DB `DEFAULT now()` on timestamps is a safety net; in practice Hibernate populates them via the audit annotations. `version` defaults to 0 for rows created outside the ORM (there shouldn't be any, but the default keeps optimistic locking sane).

## Migrations

- Migrations use **Flyway**, as **per-BC changesets** run at application startup. Each BC owns its own migration path; IAM's migrations live under the IAM module.
- This schema is the IAM module's first migration(s). Suggested split (Flyway versioned naming):
  1. `V1__iam_create_users.sql` — `users` table.
  2. `V2__iam_create_connected_sources.sql` — `connected_sources` table, FK, unique constraint, indexes.
- Encryption is an **application** concern: there is no DB-side crypto. A migration never needs the encryption key. This keeps key material out of the migration tool entirely.
- Adding a provider later (e.g. `GARMIN`) requires **no migration** — `provider` is `TEXT`, the change is in the application `Provider` enum. This is the payoff of storing enums as strings.

## What is intentionally NOT here

- **No `tokens` or `sessions` table.** Identity tokens (our JWT) are stateless; we do not persist them. Optional logout uses a `jti` blacklist in **Redis**, not Postgres (see the JWT technical note).
- **No `roles` / `authorities` table in MVP.** There is a single user type. If role-based authorization arrives later, it is an additive migration.
- **No audit-log table in MVP.** Token-access auditing is via structured logs and OTel spans (see [token-management.md](../../technical-notes/token-management.md)), not a DB table. A dedicated audit table can be added later if GDPR "what did you do with my data" reporting needs it.
